# 1. Executive Summary

- BELEGT: Die Anwendung ist aktuell eine reine Vite/React/TypeScript-SPA mit Leaflet, React-Leaflet und einem Browser-Worker. Es gibt kein Backend, keine API-Schicht und keine Datenbank.
  Belege: package.json:6-19, src/main.tsx:1-11, src/App.tsx:1-29.

- BELEGT: Primaere Verkehrsdatenquelle ist ein statischer HVV-GTFS-Feed aus dem Hamburger Transparenzportal. Daraus werden JSON-Artefakte unter public/data/hvv/ erzeugt.
  Belege: scripts/import-hvv-gtfs.mjs:9-19, scripts/import-hvv-gtfs.mjs:954-1006, src/data/hvv.ts:42-49, public/data/hvv/manifest.json.

- BELEGT: Kartenbasis ist OpenStreetMap. Optional werden Esri-Satellitenkacheln und OpenRailwayMap-Tiles angezeigt. Diese Quellen liefern keine Fahrplan-, Linien- oder Haltestellendaten.
  Belege: src/App.tsx:42-57, src/App.tsx:568-573.

- BELEGT: Geografische Abdeckung wird nicht ueber Bundeslandgrenzen, Polygone oder amtliche Schluessel modelliert, sondern ueber eine feste Bounding Box: lat 52.85..54.35, lon 8.75..11.35.
  Beleg: scripts/import-hvv-gtfs.mjs:21-28.

- BELEGT: Aktives fachliches UI-Modell ist StopPlace. Ein StopPlace gruppiert mehrere technische GTFS-Stops.
  Belege: src/domain/types.ts:10-20, scripts/import-hvv-gtfs.mjs:430-537.

- BELEGT: Liniengeometrien kommen bevorzugt aus GTFS-shapes.txt. Wenn Shapes fehlen, nutzt der Importer die Stopfolge einer repraesentativen Fahrt als Polyline-Fallback.
  Belege: scripts/import-hvv-gtfs.mjs:402-428, scripts/import-hvv-gtfs.mjs:796-847.

- BELEGT: Aktives Routing ist ein eigener clientseitiger Earliest-Arrival-Algorithmus im Browser-Worker. Er laedt reachability-index.json, rechnet ab fest 08:00, nutzt nur generische Werktags-Services und schliesst Bus/Faehre als Fahrten aus.
  Belege: src/workers/reachabilityWorker.ts:45-73, src/domain/gtfsReachability.ts:68-76, src/App.tsx:39, src/App.tsx:231.

- WIDERSPRUeCHLICH: README und docs beschreiben teils noch Seed-Router/HVV-Schaetzung als aktiven Hauptpfad und behaupten teilweise, es gebe kein Git-Repository. Der Code nutzt inzwischen StopPlaces plus Worker, und Git ist vorhanden.
  Belege: README.md:93-108, docs/CURRENT_STATE.md:48-62, docs/CURRENT_STATE.md:97-99 versus src/App.tsx:23-26, src/data/reachabilityWorker.ts:31-74, Git-Status.

- BELEGT: Die aktuell angezeigte "schnellste Fahrzeit" ist semantisch eingeschraenkt. Der Algorithmus optimiert pro Ziel auf frueheste Ankunft nach 08:00, zeigt aber travelTimeMinutes = arrivalMinutes - firstDepartureMinutes. Die anfaengliche Wartezeit ab 08:00 ist nicht enthalten.
  Belege: src/domain/gtfsReachability.ts:78-88, src/domain/gtfsReachability.ts:129-246.

- BELEGT: Durchschnittliche, mediane und P90-Fahrzeiten sind im aktiven Code nicht implementiert. Es gibt keine Statistik- oder Quantilberechnung.
  Belege: Suche nach average/mean/median/percentile/quantile/p90; src/domain/gtfsReachability.ts:238-243.

- Wichtigste Luecken: keine vollstaendige Vier-Bundeslaender-Abdeckung, keine DB, keine echte Bahnhofshierarchie, keine calendar_dates-Auswertung im Routing, keine GTFS-transfers/pathways, keine sichtbaren Fussweg-Legs, keine Durchschnitts-/P90-Metriken, keine Linienvarianten, kein Backend/API.

# 11. Fahrtzeitmetriken

## 11.1 Schnellste Fahrzeit

Status: BELEGT implementiert, aber fachlich eingeschraenkt.

Grundmenge:

- kompakte Trips im reachability-index.json
- nur weekdayMask
- mindestens 2 StopPlaces
- nicht bus, nicht faehre
- eigene raeumliche Transferkanten
- Start: StopPlace, default Hamburg Hbf
- Referenzzeit: 08:00
- kein konkretes Datum

Formel:

state(origin) = { arrival = referenceDeparture, firstDeparture = null, legs = [] }

Fuer jeden Zustand s an StopPlace i:

- Transfer:
  arrival' = s.arrival + transferMinutes
  firstDeparture' = s.firstDeparture
  legs' = s.legs

- Boarding:
  fuer jede Fahrt t mit Stop i an Position k:
    wenn departure[t,k] >= s.arrival:
      fuer jedes spaetere Ziel j:
        arrival' = arrival[t,j]
        firstDeparture' = s.firstDeparture ?? departure[t,k]
        legs' = s.legs + leg(t,i,j)

Best-State wird ersetzt, wenn:

- arrival' < existing.arrival
- oder arrival' == existing.arrival und firstDeparture' spaeter ist

Result:

- departureMinutes = firstDeparture ?? referenceDeparture
- arrivalMinutes = best.arrival
- travelTimeMinutes = arrivalMinutes - departureMinutes

Belege:

- src/domain/gtfsReachability.ts:68-76
- src/domain/gtfsReachability.ts:78-88
- src/domain/gtfsReachability.ts:129-246

Wartezeit:

- Anfaengliche Wartezeit zwischen 08:00 und erster tatsaechlicher Abfahrt ist NICHT in travelTimeMinutes enthalten.
- Umstiegswartezeit ist indirekt enthalten, wenn sie zwischen zwei Legs liegt.
- Transferkantenzeit ist in arrivalMinutes enthalten, aber nicht als Leg sichtbar.
